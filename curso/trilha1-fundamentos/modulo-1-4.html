üì• Comunicacao de Entrada (Input)

Como a skill recebe informacoes do usuario e do sistema. Definir padroes claros de entrada evita ambiguidade e erros.

üéØ Fontes de Input

  • ‚Ä¢ Mensagem do usuario: Texto livre ou comandos estruturados

  • ‚Ä¢ Resultado de tools: Output de Bash, Read, APIs
  • ‚Ä¢ Outras skills: Em workflows compostos

üìù Input Natural

Usuario fala naturalmente

‚ö° Input Estruturado

  --tipo=tecnico

üí° Dica de Design

Prefira input natural sempre que possivel. Usuarios nao querem memorizar comandos. Skill bem escrita entende "faz um PDF" e "gera documento PDF" igualmente.

üîÑ Padroes de Dialogo

Skills podem ter diferentes estilos de interacao. Escolher o padrao certo depende do caso de uso.

‚ö°

Fire & Forget

Usuario pede, skill executa

  • ‚Ä¢ Sem perguntas intermediarias
  • ‚Ä¢ Execucao imediata
  • ‚Ä¢ Bom para tarefas simples
  • ‚Ä¢ Ex: formatacao, linting
üí¨

Interativo

Dialogo para coletar info

  • ‚Ä¢ Perguntas de esclarecimento
  • ‚Ä¢ Confirmacoes antes de agir
  • ‚Ä¢ Bom para tarefas complexas
  • ‚Ä¢ Ex: setup de projeto
üîÅ

Iterativo

Refinamento continuo

  • ‚Ä¢ Mostra resultado parcial
  • ‚Ä¢ Usuario da feedback
  • ‚Ä¢ Skill ajusta e repete
  • ‚Ä¢ Ex: geracao de conteudo

üí¨ Exemplo: Padrao Interativo

üîå Comunicacao com Sistemas

Skills se comunicam com sistemas externos atraves de tools padronizadas. Entender os protocolos garante integracao robusta.

üîß Protocolos de Comunicacao

Tools Nativas

  • ‚Ä¢ Bash: stdin/stdout/stderr
  • ‚Ä¢ Read: conteudo de arquivo
  • ‚Ä¢ Write: confirmacao de escrita
  • ‚Ä¢ WebFetch: HTTP response

MCP (Externas)

  • ‚Ä¢ JSON-RPC sobre stdio
  • ‚Ä¢ Schema de request/response
  • ‚Ä¢ Erro tipados
  • ‚Ä¢ Timeouts configurados

Fluxo: Skill ‚Üí Tool ‚Üí Sistema

// 1. Skill instrui o agente
"Para verificar status do Git, use Bash com 'git status'"

// 2. Agente chama tool
Bash(command: "git status", description: "Check git status")

// 3. Tool executa e retorna
{
  "output": "On branch main\nnothing to commit",
  "exit_code": 0
}

// 4. Agente interpreta e responde
"O repositorio esta limpo, sem alteracoes pendentes."

‚úì Boas Praticas

  • ‚úì Trate erros de tools explicitamente
  • ‚úì Valide output antes de usar
  • ‚úì Use timeouts para operacoes longas

‚úó Evitar

  • ‚úó Ignorar codigos de erro
  • ‚úó Assumir que tool sempre funciona
  • ‚úó Comandos sem tratamento de falha

üìù Tom de Voz e Estilo

Skills podem definir um tom de voz especifico para suas respostas. Isso cria consistencia e personalidade.

üíº Profissional

Para contextos corporativos

üòä Casual

üìö Didatico

Para ensinar/explicar

üéØ Direto

Minimo de palavras

‚úçÔ∏è Definindo Tom na Skill 

## Estilo de Comunicacao

Ao responder, siga estas diretrizes:
- Tom: Profissional mas acessivel
- Formato: Bullet points para listas
- Comprimento: Conciso, maximo 3 paragrafos
- Evitar: Jargao tecnico desnecessario
- Incluir: Exemplos praticos quando relevante
